## [11:56] 「一键粘贴链接」扩展为「链接 / 上传图片」双模式

**用户需求**：把"一键粘贴链接"功能扩展为既可以粘贴链接，也可以直接上传服装图片。

**改动**：
1. **`app/src/components/LinkInput.tsx`**（核心重写）：
   - 顶部新增模式切换分段控件：`粘贴链接` / `上传图片`。
   - 链接模式：保持原有输入 + "识别服装" + "示例商品"。
   - 图片模式：隐藏 `<input type=file accept=image/*>` + 虚线拖拽区按钮；选图后 `FileReader` 读为 dataURL → `compressDataUrl(raw,1280,0.85)` 压缩 → 构造 `ParsedProduct`（imageUrl=压缩后 dataURL、isClothing=true、garments=[]、incomplete=true、platform="unknown"、title=文件名）→ `onParsed`。
   - 复用 `ProductCard` 已有的 `needManual`（garments 为空）分支：上传后自动进入"手动选择服装类型/颜色"流程，用户确认即试衣。
2. **`app/src/components/StepIndicator.tsx`**：步骤名 `粘贴链接` → `添加商品`。
3. **`app/src/App.tsx`**：FEATURES[2] desc 改为"粘贴购物平台链接，或直接上传服装图片"；步骤卡片标题改为"添加商品（链接 / 图片）"。

**关键链路验证**：
- 后端 `server/tryon.js` 的 `resolveToDataUrlCached` 对 `data:` 前缀直接复用（`src.startsWith("data:") return src`），故上传图片的 dataURL 作为 `product.imageUrl` 传给后端可直接当商品图使用（真试衣路径正确）。
- 本地 Canvas 预览（`tryOn`）仍用占位 SVG（`garmentDataUrl`），不使用上传图——与链接解析商品一致的限制，仅真 AI 后端路径用真实图。
- `tsc -b` 与 `vite build` 均通过（产物 `server/public/`）。

**注意**：直接上传未做 AI 自动识别服装类型，需用户在"手动选择"里指定（与平台反爬拦截时的 incomplete 流程一致）。若后续要做"上传即自动识别类型"，需新增后端视觉识别端点（当前无此前端→后端接口）。

## [12:2x] 取消「手动选择服装/颜色」，全量接入 AI 自动识别

**用户需求**：① 手动上传衣服图片后自动识别填入商品信息；② 链接与图片两种模式都取消"手动选择服装和颜色"；③ 无法识别时告知用户识别不出来、无法进行下一步。

**改动**：
1. **后端 `server/index.js`**：新增 `POST /api/recognize` 路由。入参 `{image}`（dataURL 或 URL）；未配置视觉返回 `503 NO_VISION`；调 `vision.js` 的 `recognizeProductImage` → 返回 `{ok, recognized, garment}`（recognized=false 当图非服装/模型无把握）。复用 `parse.js` 的 `makeGarment`（已 `export`）构造完整 Garment。
2. **后端 `server/parse.js`**：`makeGarment` 改为 `export`，供 index.js 复用。
3. **前端 `app/src/lib/api.ts`**：新增 `recognizeProductImageApi(dataUrl)` → POST `/api/recognize`，返回 `{recognized, garment}`。
4. **前端 `app/src/lib/parseLink.ts`**：`parseProduct` 后端解析后若 `isClothing && garments.length===0 && imageUrl 是 dataURL`，再调 `recognizeProductImageApi` 用主图补识别（视觉兜底），失败则保持空。
5. **前端 `app/src/components/LinkInput.tsx`**：图片模式上传压缩后调 `recognizeProductImageApi`；成功 → 构造带 garments 的 product 并 `onParsed`；失败 → `toast.error("无法自动识别…")` 且不进入下一步；文案去掉"请选择服装类型"。
6. **前端 `app/src/components/ProductCard.tsx`**：删除 `needManual` 手动选择服装/颜色 UI 与 `onAddManual`。改为：识别到服装 → 显示可点选列表；`isClothing 但 garments 空` → 红色错误"无法自动识别该服装信息…无法进行下一步"；非服装 → 原提示。
7. **前端 `app/src/App.tsx`**：删除 `handleAddManual` 与 ProductCard 的 `onAddManual` prop。`onParsed` 已自动 `setSelectedId(garments[0]?.id)`，识别成功即自动选中可生成；识别失败 garments 空 → `canGenerate` 自然 false。

**验证**：
- 后端 `node --check` 通过；`tsc -b` + `vite build` 均通过（产物 `server/public/`）。
- 直接单测 `recognizeProductImage`（真实服装图 dataURL）返回 `{garmentType:"tshirt", color:"#f3f4f6", region:"upper"}`，识别链路正确。
- 路由基础：health 正常、缺 image 返回 400。

**注意**：链接模式大部分靠 `parse.js` 标题关键词命中 garments；标题无命中时才走视觉兜底（需后端已配 VISION_API_KEY）。图片模式无标题，完全依赖视觉识别；若后端未配视觉模型，图片模式会提示"无法自动识别"。后台进程 `node index.js` 不会跨 Bash 调用存活，验证需在同一条命令内启动并请求。
